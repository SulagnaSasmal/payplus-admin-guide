# PayPlus Enterprise — Upgrade Procedure

This guide covers version-to-version upgrades for PayPlus Enterprise. Read the entire guide before starting. In particular, review the [breaking changes section](#breaking-changes-by-version) for the target version before proceeding.

**Audience:** System administrators responsible for maintaining PayPlus Enterprise.

> **Important:** There is no supported in-place upgrade path across more than one major version. If you are on v2.x and upgrading to v3.2, upgrade to v3.0 first, then to v3.2. Skipping major versions causes database schema migration failures.

---

## Supported upgrade paths

| Current version | Target version | Supported | Notes |
|---|---|---|---|
| 3.1.x | 3.2.x | Yes | Direct upgrade |
| 3.0.x | 3.2.x | Yes | Direct upgrade; review all 3.1 and 3.2 breaking changes |
| 2.x | 3.2.x | No | Upgrade to 3.0 first |
| 1.x | 3.2.x | No | Contact PayPlus Support for a migration plan |

If your current version is not listed, run `./bin/payplus-version.sh` to confirm it, then consult the [release notes](https://docs.payplus.io/release-notes) for your specific version.

---

## Pre-upgrade checklist

Complete every item before beginning the upgrade. An incomplete pre-upgrade state is the leading cause of upgrade failures.

### Environment

- [ ] Confirm the current version: `./bin/payplus-version.sh`
- [ ] Confirm the target installer is downloaded and its checksum matches the release notes
- [ ] Database backup is complete and validated — restore it to a test environment to confirm the backup is usable before proceeding
- [ ] A maintenance window is scheduled — upgrades require a restart (5–10 minutes downtime for single-server; use rolling upgrade for HA clusters)
- [ ] Inform operations teams and downstream systems of the maintenance window
- [ ] The upgrade is first tested in a non-production environment

### Application state

- [ ] The payment queue is empty, or in-flight payments are in an acceptable state to pause
- [ ] No bulk ACH batches are in the `SUBMITTING` state: check in **Monitoring** → **Payment Queue** → filter by `SUBMITTING`
- [ ] The audit log is flushed: `./bin/payplus-audit-flush.sh`

### Rollback preparation

- [ ] The current installation directory is backed up: `cp -a /opt/payplus /opt/payplus.backup-$(date +%Y%m%d)`
- [ ] The database backup timestamp and location are recorded here: _______________
- [ ] The rollback procedure is reviewed by at least one engineer (see [Rollback procedure](#rollback-procedure))

## Step 1: Stop PayPlus

```bash
./bin/payplus-stop.sh
```

Wait for the process to exit:

```bash
./bin/payplus-status.sh
# Expected: PayPlus is STOPPED
```

If the process does not stop within 60 seconds, check for locked transactions:

```bash
tail -100 logs/payplus.log | grep -E "SHUTDOWN|LOCKED|WAITING"
```

Do not kill the process while it is waiting for transaction locks — this can leave the database in an inconsistent state. Wait for the controlled shutdown to complete.

---

## Step 2: Back up the current installation

```bash
cp -a /opt/payplus /opt/payplus.backup-$(date +%Y%m%d)
```

This creates a copy of all configuration files, customizations, and the current application binaries. Do not skip this step even if you have a database backup — recovering from a failed upgrade sometimes requires restoring both the database and the application binaries together.

---

## Step 3: Extract the new version

```bash
cd /opt
tar -xzf ~/payplus-enterprise-3.2.x-linux-x64.tar.gz -C /opt/payplus-new
mv /opt/payplus /opt/payplus-old
mv /opt/payplus-new /opt/payplus
```

> **Do not** extract the new version over the existing installation directory. Extract separately, then swap.

---

## Step 4: Migrate configuration files

Copy your configuration files from the backup:

```bash
cp /opt/payplus-old/conf/payplus.properties    /opt/payplus/conf/
cp /opt/payplus-old/conf/payment-rails.properties /opt/payplus/conf/
cp /opt/payplus-old/conf/ofac.properties       /opt/payplus/conf/
cp /opt/payplus-old/conf/keystore.jks          /opt/payplus/conf/
cp /opt/payplus-old/conf/payplus.lic           /opt/payplus/conf/
```

**Do not** copy `log4j2.xml` or `payplus-install.properties` from the old installation — these are regenerated by the upgrade and copying old versions can mask upgrade errors.

After copying, review the [breaking changes](#breaking-changes-by-version) for the target version and apply any required configuration changes before proceeding.

---

## Step 5: Run the database migration

```bash
cd /opt/payplus
./bin/payplus-migrate.sh --from 3.1.x --to 3.2.x --config conf/payplus.properties
```

The migration script applies incremental SQL change scripts from the `sql/migrations/` directory. It is idempotent — you can run it again if it is interrupted, and it will continue from where it stopped.

Expected output:

```
[INFO] Current schema version: 3.1.12
[INFO] Target schema version:  3.2.4
[INFO] Applying migration 3.1.12 → 3.1.13... OK (1.2s)
[INFO] Applying migration 3.1.13 → 3.2.0...  OK (8.4s)
[INFO] Applying migration 3.2.0  → 3.2.1...  OK (0.3s)
[INFO] Applying migration 3.2.1  → 3.2.4...  OK (2.1s)
[INFO] Schema migration complete. Version: 3.2.4
```

If any migration fails, the script rolls back the failed step and exits with a non-zero status. Check `logs/migrate-YYYYMMDD-HHMMSS.log` for details.

---

## Step 6: Start PayPlus and validate

```bash
./bin/payplus-start.sh
tail -f logs/payplus.log
```

Wait for the startup confirmation:

```
[INFO] PayPlus Enterprise 3.2.x started on port 8443
[INFO] Schema version: 3.2.4 — OK
```

### Post-upgrade validation checkpoints

Complete each checkpoint before declaring the upgrade successful:

- [ ] Administration console loads at `https://payplus-app.internal:8443/admin`
- [ ] Console version string shows `3.2.x`
- [ ] **Administration** → **System** → **License**: status is Valid, modules match the pre-upgrade state
- [ ] **Monitoring** → **System Health**: all connectors show green status
- [ ] Submit a test payment through **Tools** → **Payment Tester**
- [ ] Verify one report runs correctly: **Reports** → **Daily Settlement Report**

---

## Rollback procedure

Use this procedure if the post-upgrade validation fails and the system cannot be restored to a working state by configuration changes alone.

1. Stop PayPlus: `./bin/payplus-stop.sh`
2. Swap the installation directories back:
   ```bash
   mv /opt/payplus /opt/payplus-failed
   mv /opt/payplus-old /opt/payplus
   ```
3. Restore the database from your pre-upgrade backup (procedure depends on your DBA tooling — coordinate with your DBA)
4. Start the previous version: `./bin/payplus-start.sh`
5. Confirm the console loads and shows the previous version number
6. File a support ticket at support.payplus.io with the migration log from `logs/migrate-*.log`

> **Important:** The database schema must match the application version. You cannot run the v3.1 application binaries against a v3.2 schema, or vice versa. Always restore the database to the pre-upgrade backup when rolling back.

## Breaking changes by version

Review all entries between your current version and your target version.

### v3.2

| Component | Change | Action required |
|---|---|---|
| **ACH connector** | The `ach.gateway.effective_date_offset` parameter now accepts `0` (same-day ACH) in addition to `1` and `2`. No behavior change for existing values. | None unless you want to enable same-day ACH. |
| **Dual control** | The threshold parameter was renamed from `workflow.dual_approval.threshold` to `workflow.dual_control.threshold_usd`. Old parameter name is ignored. | Rename the parameter in `payplus.properties` before upgrading. |
| **Audit log** | Audit log entries now include a `session_id` column. If you have custom reports that query the `AUDIT_LOG` table by column position, they will break. | Update custom reports to use column names, not ordinal positions. |
| **SSO** | The `sso.idp.metadata_url` parameter is now required when SSO is enabled. Previously, inline metadata via `sso.idp.metadata_xml` was sufficient. | If you use SSO with inline metadata, expose the metadata via URL and add `sso.idp.metadata_url`. |
| **Minimum Java** | Java 11 is no longer supported. Java 17 is the new minimum. | Upgrade Java before upgrading PayPlus. |

### v3.1

| Component | Change | Action required |
|---|---|---|
| **Database pool** | The `db.pool.max_lifetime_ms` parameter was added to address connection staleness on Oracle RAC. Default: `1800000` (30 minutes). | None for standard deployments. If you see `ORA-02396` errors after upgrade, the default value resolves them. |
| **SWIFT** | ISO 20022 pacs.008 messages are now generated by default for Fedwire and SWIFT. MT103 remains available behind the `feature.swift.use_mt_format=true` flag. | No action required unless a downstream system requires MT103 exclusively — set the flag. |
| **Session management** | Concurrent session limit per user was introduced. Default: 3 sessions. Configure with `app.session.max_concurrent`. | Inform users who rely on multiple simultaneous sessions (unusual but documented in support cases). |

## High-availability cluster rolling upgrade

For HA cluster deployments, use the rolling upgrade procedure to avoid downtime:

1. Remove node 2 from the load balancer.
2. Run Steps 1–6 (stop, backup, extract, migrate config, migrate database, start) on node 2.
3. Validate that node 2 is running correctly.
4. Add node 2 back to the load balancer.
5. Remove node 1 from the load balancer.
6. Run Steps 1–3 and 5–6 on node 1 — skip Step 4 (database migration) because it was already applied on node 2.
7. Add node 1 back to the load balancer.

> **Note:** The `payplus-migrate.sh` script is idempotent. Running it a second time on a schema that is already at the target version is safe, but produces a log entry stating that no migrations were applied.
